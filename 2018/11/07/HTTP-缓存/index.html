HTTP 缓存

2018-11-07

最近学习pwa，了解了server worker的缓存能力，但对于HTTP缓存还是一知半解，借此机会学习并记录。

缓存主要指的是代理服务器或客户端本地磁盘内保存的资源副本。本文主要涉及的是客户端（浏览器）内的缓存。
要想了解HTTP缓存，首先应该从宏观方面对其工作流程有所了解，然后再具体到流程的具体步骤，对其细节进行分析。

HTTP缓存流程

首先来看《HTTP权威指南》里的这张“缓存GET请求的流程图”，本张图基本描绘出了GET请求的缓存过程，其中，整个流程又可以分为两个方面：强缓存与协商缓存。

强缓存

首次发起请求时，浏览器会在Response Header中写入缓存新鲜时间。当请求再次发出时，如果浏览器判断缓存新鲜，将会直接从缓存中回去资源，而不再与服务器发生通信。

协商缓存

协商机制下，浏览器向服务器询问相关信息，进而判断资源是否发生变动，如果变动了，则返回新的资源实体，如果没有变，则继续使用已有缓存。

具体细节分析

由上述流程可以看出，缓存主要分为强缓存及协商缓存。只要我们将强缓存与协商缓存的具体流程弄明白，也就明白了缓存的过程。HTTP缓存与首部有着密不可分的关系，只要清楚了各个首部的含义，自然也就清楚了明白了缓存的过程。

强缓存策略

当对资源进行初次请求的时候，服务器会在Response Headers中带上Expires、Cache-Control（可能两个都有，也可能只有其一），
这两个首部都可以存储该资源的过期时间。当浏览器再次访问该资源时，浏览器会对比这两个字段的值，判断资源是否过期，从而判断是否向服务器发起请求。

Expires在HTTP/1.0提出，其值为一个绝对时间，像Wed, 07 Nov 2018 10:42:14 GMT这样的时间格式。
这会造成一个潜在的隐患，即服务器时间与本地系统时间不一致时，判断缓存是否过期的时候可能出现差错。
如何解决这个问题？HTTP/1.1提出了Cache-Control字段。

Cache-Control中，用来存储过期时间的格式为Cache-Control: max-age=86400，这是一个相对的日期，因此就没有了时间不一致的隐患。

如何计算缓存寿命？

对于含有特定头信息的请求，会去计算缓存寿命。比如Cache-control: max-age=N的头，相应的缓存的寿命就是N。通常情况下，对于不含这个属性的请求则会去查看是否包含Expires属性，通过比较Expires的值和头里面Date属性的值来判断是否缓存还有效。如果max-age和expires属性都没有，找找头里的Last-Modified信息。如果有，缓存的寿命就等于头里面Date的值减去Last-Modified的值除以10（注：根据rfc2626其实也就是乘以10%）。

https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Caching_FAQ

Cache-Control常用的其他值:

  • Cache-Control: private 默认值为private，表示该响应是专用于某单个用户的，中间人不能缓存此响应，该响应只能应用于浏览器私有缓存中。

  • Cache-Control: public 表示该响应可以被任何中间人（比如中间代理、CDN等）缓存。

  • Cache-Control: no-store 禁止缓存，每次请求都要向服务器重新获取数据。

  • Cache-Control: no-cache 标识为no-cache的响应可以存储在缓存中，只是将缓存提供给客户端使用之前，需要与原服务器进行新鲜度再验证。

  • Cache-Control: must-revalidate

Cache-Control: max-age=0, no-cache, must-revalidate区别。
